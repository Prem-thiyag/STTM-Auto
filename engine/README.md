# engine

A standalone Python application that executes SQLX ETL projects generated by
the `sqlx-etl-generator` Claude Code Skill (`.claude/skills/sqlx-etl-generator/`).

**Independent of Claude.** Nothing in this package imports an LLM, reads a
source document (`source_schema.md`, `target_schema.md`, the STTM workbook,
`user_defined_functions.md`, `folder_hierarchy.md`), calls a specialist, or
reasons about ETL logic. It reads metadata the skill already generated, and
executes SQL. If you're looking for where mappings get decided, that's the
skill's `Mapping Resolver` — not here. This package performs zero reasoning.

## Overall architecture

```
metadata/execution/execution_plan.json ──▶ planner.py ──▶ ordered ExecutionStep[]
                                                                 │
definitions/<TABLE>/<stage>.sqlx ──▶ parser.py ──▶ SqlxFile(config, sql) ◀┘
                                                                 │
                                                          executor.py (Engine)
                                                                 │
                                          ┌──────────────────────┼──────────────────────┐
                                          ▼                      ▼                      ▼
                                   database.py            postgres.py            logger.py
                                (DatabaseExecutor ABC)  (PostgresExecutor)   (ExecutionLogger)
```

## Module responsibilities

| Module | Responsibility |
|---|---|
| `models.py` | Typed dataclasses for every value that flows between modules: `Stage`, `SqlxConfig`, `SqlxFile`, `ExecutionStep`, `ExecutionPlan`, `ExecutionRecord`, `ConnectionConfig`. No behavior, just shape. |
| `exceptions.py` | `EngineError` and five subclasses — `SQLXParseError`, `ExecutionPlanError`, `DatabaseConnectionError`, `ExecutionFailure`, `ValidationError`. Every error this package raises is one of these; nothing raises a bare `Exception`. |
| `utils.py` | Generic helpers with no SQLX/Postgres/plan-specific logic: `now_iso()`, `sha256_of_text()`, `find_matching_brace()` (brace-depth + string-aware, used by `parser.py`). |
| `parser.py` | Loads one `.sqlx` file, locates and JSON-parses its `config { ... }` block, validates required properties, and returns `SqlxFile(config, sql)`. Never touches the SQL past extracting it as a string. |
| `planner.py` | Loads `metadata/execution/execution_plan.json`, validates it's structurally sound and that its own step order is already a valid dependency order (every `depends_on` exists and appears earlier). Never computes or reorders anything — an invalid plan is a hard error, not something to fix here. |
| `database.py` | The `DatabaseExecutor` abstract base class (`connect`/`execute`/`begin_transaction`/`commit`/`rollback`/`close`) plus `MockDatabaseExecutor`, an in-memory implementation for tests. No implementation here or elsewhere may import `executor.py` — the dependency direction is one-way. |
| `postgres.py` | `PostgresExecutor(DatabaseExecutor)`, backed by `psycopg2`, plus `resolve_connection_config()` for turning `.mcp.json` / environment variables into a `ConnectionConfig` without ever hardcoding a credential. |
| `logger.py` | `ExecutionLogger` — appends one `ExecutionRecord` per executed stage to `metadata/execution/engine_execution_log.json`. |
| `executor.py` | `Engine` — the orchestrator. Wires `planner.py` + `parser.py` + a `DatabaseExecutor` factory + `ExecutionLogger` together and implements the execution flow, transaction model, and stop-on-failure rule below. Also the CLI (`main()`, invoked via `__main__.py`). |

## Execution flow

```
load execution plan (planner.py)
  │
  ▼
for each step, in the plan's own order:
  │
  ├─ read the .sqlx file named by the step, parse its config block (parser.py)
  ├─ get a DatabaseExecutor for the config block's `database` (injected factory)
  ├─ connect() → begin_transaction()
  ├─ execute(sql)
  │     ├─ success → commit() → log a success record → close() → next step
  │     └─ failure → rollback() → log a failure record → close()
  │                  → raise ExecutionFailure → STOP (no further step runs,
  │                    not even a later stage of the same table)
```

This matches the project's ETL shape directly: each target table's `read.sqlx`
→ `process.sqlx` → `write.sqlx` are three separate steps in the plan, so "stop
on failure, don't continue to the next table" falls out of "stop on failure,
period" — the planner's step order already interleaves tables and stages
correctly (see `docs/examples/generated-project/metadata/dependency/dependency_graph.json`
in the skill for why `FACT_ORDER.process` depends on `DIM_CUSTOMER.write`, not
just its own `FACT_ORDER.read`).

## SQLX format

Every `.sqlx` file the skill generates now opens with a machine-readable
config block, then a blank line, then SQL:

```sql
config {
    "stage": "read",
    "buildspec": "metadata/build/DIM_CUSTOMER.buildspec.json",
    "database": "intermediate_db",
    "version": "1.0"
}

CREATE TABLE IF NOT EXISTS stg_dim_customer ( ... );
...
```

`parser.py` parses **only** the `config` block — locates the `config`
keyword, finds the matching closing brace (via `utils.find_matching_brace`,
which is string-literal-aware so a stray `}` inside a quoted value can't
truncate the block early), and `json.loads()`s the substring between them.
Everything from the first non-blank line after the closing `}` onward is
handed to the database driver completely unmodified — no tokenizing, no
rewriting, no validation of the SQL itself. `stage`, `buildspec`, and
`database` are all required; a `.sqlx` file missing any of them, or with an
unparsable config block, fails with a descriptive `SQLXParseError` naming the
exact file and problem. `version` is optional — present on every file the
current generator writes, but a `.sqlx` file without one still parses
(`SqlxConfig.version` is `None` in that case).

See the skill's `references/sqlx-syntax-guide.md` for the full rationale
(including why `database` is in the config block at all — it's not derivable
from the buildspec alone, since the intermediate database's name is a
renderer parameter, not a stored field).

## Execution plan is declarative

`metadata/execution/execution_plan.json` carries no shell command or MCP
invocation string — only `step_id`, `table`, `stage`, `database`, `file`, and
`depends_on` (`schemas/execution_plan.schema.json` in the skill). This is
deliberate: the plan is shared between the human-facing `Execute` plan (which
formats its own display command from a step's fields) and this engine (which
never needs a command string at all — it executes a step's `.sqlx` file's SQL
directly). See the skill's `docs/ASSUMPTIONS.md` "`execution_plan.json` is
declarative" for the full rationale, including why this changed from an
earlier design that did store a resolved command per step.

## Declared vs. observed database

A step's target database is recorded in **two** places by design —
`execution_plan.json`'s `step.database`, and the step's own `.sqlx` file's
`config.database` — rather than one owning it exclusively. `Engine._run_step`
cross-checks the two agree before doing anything else for that step, raising
`ValidationError` (naming both values and the file) if they don't, rather than
silently trusting whichever it happened to read first. This catches a plan or
`.sqlx` file that was hand-edited out of sync with the other — something a
single-source-of-truth design would have no way to detect at all, since
there'd be nothing to compare against.

## Transaction model

One `.sqlx` file = one transaction = one `DatabaseExecutor` instance, used
once and discarded. `connect()` and `close()` bracket every single step, not
the whole run — there is no shared, long-lived connection pooled across
steps. This is a deliberate simplicity/robustness tradeoff: a stage that
fails can never leave a transaction open or a connection in an ambiguous
state for the next stage to inherit, at the cost of reconnecting once per
`.sqlx` file (six times for a two-table project). If this becomes a real
performance concern at scale, connection pooling is a `postgres.py`-only
change — `Engine` and `DatabaseExecutor` don't know or care how a connection
was obtained.

## Logging model

`metadata/execution/engine_execution_log.json` — **not**
`metadata/execution/execution_log.json`. That file already belongs to the
`sqlx-etl-generator` skill's `Execute` plan (per its `ARCHITECTURE.md`
ownership table) and records something different and much smaller: one entry
per *human* confirming they manually ran a command. This engine's log is
machine-observed telemetry from an actual automated run:

```json
{
  "entries": [
    {
      "table": "DIM_PATIENT",
      "stage": "read",
      "start_time": "2026-01-01T00:00:00.000000Z",
      "end_time": "2026-01-01T00:00:00.041000Z",
      "duration_seconds": 0.041,
      "status": "success",
      "rows_affected": 6,
      "sql_hash": "sha256:...",
      "error_message": null
    }
  ]
}
```

Append-only in effect: each `ExecutionLogger.record()` call reads the whole
file, appends one entry, and writes it back. There's no expectation of
concurrent writers — one `Engine.run()` is one process executing steps
strictly in sequence, never in parallel.

## Database abstraction

`DatabaseExecutor` is deliberately minimal — six methods, no query builder,
no ORM, no schema introspection. `PostgresExecutor` is the only production
implementation today; `MockDatabaseExecutor` (in `database.py`) exists purely
for tests, records every call it receives in order, and can be told to raise
on any SQL containing a given substring (`fail_on=...`) to exercise the
rollback/stop-on-failure path deterministically without a real database.

## Connection configuration

Never hardcoded. `postgres.resolve_connection_config(dbname)` tries, in
order:

1. **`.mcp.json`** (repository root) — looks for an `mcpServers` entry whose
   name contains "postgres" and reads `PGHOST`/`PGUSER`/`PGPASSWORD`/`PGPORT`
   (or `POSTGRES_*`) out of its `env` block. Absent, empty, or malformed
   `.mcp.json` falls through silently to step 2 — this is a preference, not a
   requirement.
2. **Standard environment variables** — `PGHOST`, `PGPORT` (default `5432`),
   `PGUSER`, `PGPASSWORD`.

Raises `DatabaseConnectionError` — never guesses, never returns a
partially-filled config — if neither source has enough information. You can
always bypass resolution entirely and construct a `ConnectionConfig` yourself
for `PostgresExecutor(config)`.

## How to run

```
pip install -r engine/requirements.txt
python -m engine [project_root]        # defaults to output/
```

`project_root` must contain `definitions/`, `metadata/`, and `bootstrap/` —
i.e. it's the directory the skill's `Generate` plan wrote to (`output/` by
default; see the skill's `plans/generate.md`).

Programmatic use:

```python
from pathlib import Path
from engine import Engine, PostgresExecutor, resolve_connection_config

def factory(database: str):
    return PostgresExecutor(resolve_connection_config(database))

engine = Engine(Path("output"), factory)
outcomes = engine.run()  # raises ExecutionFailure on the first failing step
```

## Testing

```
pip install -r engine/requirements.txt pytest
python -m pytest engine/tests/ -v
```

42 tests, all passing as of this writing, covering `parser.py`, `planner.py`,
`executor.py`, `logger.py`, `utils.py`, and the pure-logic parts of
`postgres.py` (connection-config resolution). Three tests in
`engine/tests/test_executor.py` run the full `Engine` against the *actual*
generated project this repository's `output/` currently contains (real
schema, real STTM-derived buildspecs, real rendered SQLX, including a real
`GENERATED`/`ROW_NUMBER` surrogate-key column) using `MockDatabaseExecutor` —
one proving a clean six-step run end-to-end, one proving that a simulated
failure on `DIM_PATIENT/read.sqlx`'s `udf_full_name(...)` call rolls back,
logs the failure, raises `ExecutionFailure`, and stops before any other step
(including `DIM_PATIENT`'s own later stages) runs at all, and one proving the
plan/config-block database consistency check (see "Declared vs. observed
database") raises `ValidationError` rather than silently trusting one side.

**`PostgresExecutor.connect()` is not verified against a live database as of
this milestone**, by explicit decision — live PostgreSQL validation is a
separate, later milestone once the full skill (Generate → Review → Execute →
Clean) is finished end-to-end. `postgres.py`'s connection-resolution logic
*is* unit-tested (`engine/tests/test_postgres.py`); only the actual socket
connection and query execution against a real server are deferred.

## Extending with additional databases

`DatabaseExecutor` is the seam: implement it for another engine (MySQL,
Snowflake, BigQuery — whatever a future `sqlx-etl-generator` dialect targets,
see the skill's `docs/EXTENSION_POINTS.md` "Adding a new SQL dialect") and
pass a factory that returns your implementation into `Engine`. Nothing in
`executor.py`, `planner.py`, or `parser.py` needs to change — none of them
import `psycopg2` or know PostgreSQL exists; only `postgres.py` does. If the
new dialect's SQLX output also needs a different config-block shape, that's a
`parser.py` change, made once, not a per-database special case scattered
through the orchestrator.

## Known gaps (found while building against real generator output)

- **Fixed, no longer a gap:** the skill's `sttm.schema.json` originally had
  no way to represent a "generated column, no source" mapping (e.g. a
  surrogate key produced by `ROW_NUMBER()`), and its deterministic Excel
  parser (`scripts/parse_sttm.py`) hard-failed on an STTM row with a blank
  source table/column. A repository refinement pass added `GENERATED` (and
  `DEFAULT`/`SEQUENCE`/`UUID`) as first-class sourceless mapping kinds at
  both the `sttm.json` and buildspec IR layers (see the skill's
  `docs/ASSUMPTIONS.md` "GENERATED columns are a first-class mapping type").
  `output/`'s `DIM_PATIENT.patient_key` and `FACT_PATIENT_VISIT.visit_key`
  now use `transformation: "GENERATED"` with a `generator:
  {"type": "ROW_NUMBER", ...}` object, not the `EXPRESSION`-with-null-source
  workaround this section used to describe.
- `input/user_defined_functions.md` still declares UDF signatures but not
  SQL bodies, so `bootstrap/db/03_intermediate/create_udfs.sql` in `output/`
  is empty (by design — the generator never fabricates SQL, see the skill's
  `docs/ASSUMPTIONS.md`). This is no longer a silent gap: `scripts/gen_bootstrap.py`
  now emits an explicit `WARNING:` (stderr, and in the generated file's own
  header comment) naming exactly what will fail. Running the engine against
  a live database with this `output/` as-is will genuinely fail on
  `DIM_PATIENT/read.sqlx`'s `udf_full_name(...)` / `udf_calculate_age(...)`
  calls until real UDF bodies are added to that document and `Generate` is
  re-run — `engine/tests/test_executor.py`'s failure-path test exercises
  exactly this scenario (simulated, via `MockDatabaseExecutor`).
